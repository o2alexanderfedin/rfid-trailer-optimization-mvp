# v1.2 Hub Detail Panel — Codebase Grounding

> This document grounds the v1.2 **Hub Detail panel** feature — clicking a hub icon on the map opens a right-rail panel showing the trailers currently at that hub, their operational status, cargo/load plan (rear→nose LIFO order), elapsed dwell, next hub, and estimated ETA. It was produced by an **adversarial multi-agent codebase analysis**: six recon agents mapped data availability per dimension, a synthesis agent proposed requirements/phases, and a verification pass issued **adversarial verdicts** that downgraded several optimistic "available/derivable" claims. The tables below reflect the *verified* truth after correction — read them as the source of ground truth for phase planning, not the raw synthesis.

## Delivery Decision

**Partially client-side, but a new REST endpoint is required.** (`synth.clientSideFeasible`.)

The ws channel **cannot** carry the heavy hub detail: ws `HubState` carries only metric buckets `{ id, volumeBucket, slaRiskBucket, congestionBucket }` (`envelope.ts` L62-67) — **no trailer references**. The following are server-side reconstructions never streamed: (1) the trailers-currently-at-this-hub list, (2) each trailer's `assigned_package_ids` + reconstructed rear→nose plan / instructions / utilization (via `planLoad`, `plan-detail.ts`), (3) `arrivedAtMs` (only in the event log / `audit_timeline`), and (4) next-hub derivation from package `nextUnloadHubId`.

**Recommended split (KISS, no ws schema change):**
- **New `GET /api/hubs/:id/detail` (REST-on-click)** supplies the heavy DTO: presence list + per-trailer load plan + utilization + `arrivedAtMs` + `nextHubId` + estimated ETA + `dock_door_id`.
- **The ws envelope supplies the live deltas already streamed:** `WsEnvelope.simMs` (drives live-ticking elapsed dwell and ETA countdown client-side), in-transit `TrailerKeyframe.etaMs`, trailer coloring state, and `exceptionsOpen` deltas filtered by `entityId`.

So: the endpoint covers **presence + load plan + arrival time + next hub**; the client store covers **the live clock and alerts**.

## Risks & Open Questions

### Risks (`synth.risks`)
1. **Refined operational status is genuinely missing.** No `UnloadStarted`/`LoadStarted` events exist (`schemas.ts` L104-120). Any loading/unloading/staged-idle label in v1.2 is a heuristic over `status` + `dock_door_id` + `assigned_package_ids` and can be wrong; presenting it as authoritative risks misleading the demo audience.
2. **ETA for staged trailers is missing/estimated.** The only persisted ETA (ws `TrailerKeyframe.etaMs`) applies solely to already-in-transit trailers, and `TrailerDeparted` carries no planned departure. Any hub-panel ETA for a parked trailer is an estimate from `TimingConfig` distributions, not a committed schedule — the `'late'` coloring is intentionally never emitted for the same reason.
3. **Dwell correctness trap.** Elapsed dwell must come from `audit_timeline` `TrailerArrivedAtHub`, **not** `trailer_state.last_event_at` (which advances on later events like `TrailerDocked`). Using `last_event_at` would silently under-report dwell — flagged across multiple recon dimensions.
4. **Per-trailer plan reconstruction cost.** Reconstructing load plans for every trailer at a busy hub multiplies per-trailer `planLoad` cost (O(n_trailers) per panel open); for the center hub (higher occupancy) this could be noticeable click latency without batching or a shared reconstruction helper.
5. **Next-hop point-in-time inconsistency.** The authoritative next-hub route (`TwinStop[]`) lives only in the in-memory `TwinSnapshot` per epoch and is not persisted; the panel re-derives next hop from package `nextUnloadHubId`, which can diverge from what the optimizer would actually plan at the next epoch.
6. **VIZ-05 wire-contract regression.** Adding `utilization` to `TrailerPlanDto` touches the working VIZ-05 contract; a careless change could regress the `TrailerDetail` panel — make it additive-only and preserve existing tests.

### Open Questions (`synth.openQuestions`)
1. **Operational status:** Accept an INFERRED loading/unloading/staged-idle label (heuristic, clearly marked estimated), or build true phase tracking via new domain events (`UnloadStarted`/`UnloadCompleted`/`LoadStarted`) — a larger, separate scope?
2. **ETA for parked trailers:** Is an ESTIMATE (arrival + expected dwell + expected transit from `TimingConfig`, labelled "est.") acceptable for the demo, or must it be a committed schedule (which requires persisting plan departure times that don't exist today)?
3. **Delivery channel:** New REST `GET /api/hubs/:id/detail` (fetch-on-click, matches VIZ-05, keeps ws small) — *recommended* — or extend ws `HubState` to carry the trailers-at-hub list (larger ticks, always-live)?
4. **Load-plan scope in the panel:** Full rear→nose + instructions + explanation for every trailer (heavier, full VIZ-05 reuse), or a compact summary (utilization + package count + next hub) with click-through to the existing trailer panel for the full plan?
5. **Index decision:** Add the `current_hub_id` btree index (HUBQ-02), or defer it because the demo hub count is small enough that the full-table scan is tolerable?

## Verification Notes

The adversarial verification pass issued **9 verdicts** (2 `confirmed`, 7 `downgraded`). What it **confirmed** vs **corrected**:

**Confirmed as claimed:**
- Trailers-at-hub presence IS derivable from `trailer_state WHERE current_hub_id = :hubId`, and the missing `current_hub_id` index is real (full-table scan without HUBQ-02).
- The load plan IS derivable via the exact VIZ-05 `plan-detail.ts` reconstruction pattern.

**Corrected / downgraded (reflected in the table above):**
- **Utilization** is **slice-based** `Σ usedVolume / Σ capacityVolume` over `plan.slices` (per-slice capacity ≈ `maxBlockVolume`, default 30) — **not** a flat `block.totalVolume / 50`. The multi-slice model fills zones partially, so the denominator is zone-wise.
- **Dwell** must use `audit_timeline` `TrailerArrivedAtHub.occurred_at`, **not** `trailer_state.last_event_at`. The read helper (`readTrailerAuditTimeline`) exists but is **not yet a REST endpoint** — it needs exposing.
- **Next-hub** is **not** pushed to the client over ws and is **not** trivially available; it needs explicit API plumbing (reconstruct via `buildRoute`). The synthesis's "no persistence needed / reconstruct on demand" understated the missing API surface.
- **Exceptions:** ws `ExceptionItem.entityId` carries **only `trailerId`** — there is **no hubId dimension** in the stream. Client-side filtering works for trailer-scoped alerts only; the "(and/or hubId)" claim was **wrong**. Full hub coverage needs a server-side `WHERE hub_id = ?` query.
- **`dock_door_id`, coarse status:** raw values are fully populated and persisted, but the **API surface does not expose `dock_door_id`** today — additive DTO work required.
- **Hub basics / hub-click:** data is on the client, but the verdict stressed this is **implementation work, not a data-fetching gap** — no `HubDetail` component, no `onHubSelect` callback, no `selectedHubId` state, and the `MapView` click handler reads only `trailerId` (L164-181). "No new map infrastructure" is accurate, but it is **not** a "frontend-only read-only derivation."

**Net takeaway for phase planning:** the data exists or is derivable, but the gaps are **API plumbing** (new `/hubs/:id/detail` endpoint, exposed arrival-time query, `dock_door_id`/`util`/`nextHubId` DTO fields, `current_hub_id` index) and **UI wiring** (hub-click → new `HubDetail` panel), plus two explicitly **estimated/heuristic** signals (refined status, parked-trailer ETA) that must be labelled as such in the demo.
