# Phase 27 — UI Design Contract

> Visual and interaction contract for the THREE scale-visualization requirements of Phase 27
> (VIZ-15 declutter, VIZ-16 tier hierarchy, VIZ-17 suggestion overlay). The PERF-02/03/04 and
> P27-A/P27-B items are non-visual and are out of scope for this contract.
>
> This is INCREMENTAL work on the EXISTING OpenLayers + React + Vite map. The established design
> system is a hand-authored dark-slate operator theme (plain CSS, no Tailwind/shadcn) plus a
> zero-allocation cached-OL-`Style` convention (`coloring.ts`/`stopColoring.ts`/`inductionColoring.ts`).
> Everything below EXTENDS those conventions — it does not redesign them. Design decisions were
> largely locked in `27-CONTEXT.md` (Area 3); those are treated as decided and not re-asked.
>
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | none (hand-authored CSS + cached OpenLayers `Style` objects) |
| Preset | not applicable |
| Component library | none (plain React 19 function components, BEM-ish CSS class names) |
| Icon library | none — Unicode/emoji glyphs drawn into OL `Text` styles (🏭 hub, 🚛 trailer, `+` induction); React panels use text/badges, no icon font |
| Font | `system-ui, -apple-system, sans-serif` (set on `html,body,#root` in `index.css`); emoji glyphs via `"Apple Color Emoji","Segoe UI Emoji","Noto Color Emoji"` |

**shadcn gate:** N/A. The project is Vite + React but deliberately uses no component library — the map
is canvas/WebGL-rendered by OpenLayers and the surrounding panels are a small bespoke CSS system. Introducing
shadcn for a 3-requirement incremental viz pass would fragment the established operator theme. `Tool: none`.

**Hard rendering convention (non-negotiable — extend, never violate):**
- One reused `ol/source/Vector` per logical layer for the map's lifetime; **never** `source.clear()` / rebuild.
- Update a feature's appearance by mutating a feature property (`feature.set("tier", ...)`) so OL re-invokes the
  cached `StyleFunction` on the next render. **Never** `feature.setStyle(new Style(...))` per feature/frame.
- Pre-allocate ONE `Style` per discrete bucket/tier/state at module load → zero per-frame allocation (P10 / T-01-24).
- Static topology (hubs/routes geometry incl. the new `kind`/`tier`/`isBackbone` fields) is fetched ONCE via REST
  on map init; the ws `tick` carries only id-keyed metric buckets + transient arrays. Suggestions are transient,
  tick-only (mirror `inductionEvents`), and **never** on the snapshot (Pitfall-7 rule).

## Spacing Scale

The existing CSS uses `rem`-based spacing (root font-size **14px**, so `0.5rem ≈ 7px`). New panel CSS
MUST reuse the EXISTING tokens below (taken verbatim from `index.css`) for visual consistency — do NOT
introduce a parallel px scale in the React panels. The 4px-multiple table is the conceptual reference;
the concrete values new code must emit are the existing-rem column.

| Token | Existing rem (use this) | ≈ px @14px root | Usage in this phase |
|-------|-------------------------|-----------------|---------------------|
| xs | `0.15rem`–`0.25rem` | 2–4px | Inline gaps inside a suggestion row header, badge padding |
| sm | `0.4rem`–`0.5rem` | 6–7px | Suggestion-row vertical padding, header gap (matches `.alert-feed__entry`) |
| md | `0.6rem`–`0.75rem` | 8–11px | Suggestion-row horizontal padding (matches `.alert-feed__entry` `0.5rem 0.75rem`) |
| lg | `1rem` | 14px | Section padding / empty-state padding (matches `.alert-feed__empty`) |

**Map-marker spacing (OpenLayers px — separate from CSS rem; these are screen px, intentionally NOT rem):**

| Token | Value | Usage |
|-------|-------|-------|
| Cluster distance | **40px** | `ol/source/Cluster` `distance` — radius within which hub points group into one cluster (VIZ-15) |
| Cluster minDistance | **20px** | `ol/source/Cluster` `minDistance` — smallest separation between rendered clusters (must be ≤ distance) |
| Center marker radius | **20px** | Tier 1 hub disc (large) — VIZ-16; +4px over the existing 16px spoke radius |
| Spoke marker radius | **12px** | Tier 2 hub disc (small) — VIZ-16; −4px under the existing 16px default so centers dominate |
| Suggestion overlay marker radius | **13px** | VIZ-17 transient advisory marker (between induction 14 and spoke 12) |

Exceptions: marker radii and cluster distances are screen-pixel OpenLayers values, not the 4px CSS rem
scale — this is the established split in the codebase (see existing `MARKER_RADIUS = 16`, `INDUCTION_RADIUS = 14`).

## Color

The established palette is a dark-slate operator theme. The map's semantic encodings are already
saturated (hub green→red volume ramp, blue route-load ramp, warm SLA-risk ramp, purple induction,
teal consolidation). The Phase-27 additions MUST slot into the unused semantic space and the existing
60/30/10 split below — do NOT recolor existing encodings.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f172a` (slate-900) | App + map chrome + rail background; the OSM basemap fills the map canvas |
| Secondary (30%) | `#1e293b` (slate-800) / borders `#334155` (slate-700) | Rail section headers, KPI/suggestion cards, dividers |
| Accent (10%) | `#1d4ed8` / `#60a5fa` (blue) | Active tab, selected/interactive emphasis, the new **Suggestions toggle ON state** |
| Destructive | `#dc2626` / `#ef4444` (red) | Reject suggestions, exception-count badge, high-severity exception border |

**Accent reserved for (explicit):** the active right-rail tab; the Suggestions overlay toggle in its ON state;
hover/focus emphasis on interactive controls; the route-load blue ramp (existing). It is NOT used as a generic
"all clickable things are blue" rule.

### VIZ-16 tier hierarchy — distinct visual tier (centers vs spokes vs backbone legs vs spoke legs)

Tier separation is encoded by **size + weight + a tier outline**, NOT by hue (hue is already owned by the
volume/load/risk ramps). A center and a spoke at the same volume bucket keep the SAME fill ramp color; the
center reads as a center by being larger with a distinct ring. This preserves every existing metric encoding.

| Element | Tier signal | Concrete value | Rationale |
|---------|-------------|----------------|-----------|
| Regional center hub (Tier 1) | Large disc + gold/amber ring | radius **20px**, stroke `#f59e0b` (amber) width **3px**, keep `volumeBucket`/`dutyBucket` fill ramp | Centers are the structural backbone nodes — must dominate at zoom 4 |
| Spoke hub (Tier 2) | Small disc + white ring | radius **12px**, stroke `#ffffff` width **2px** (existing), keep fill ramp | Spokes recede so 80–130 of them read as a field, not noise |
| Backbone leg (heavy) | Thick, brighter, slightly translucent | stroke `#cbd5e1` (slate-300) width **4px**, `opacity 0.9`, keep the load/risk ramp where present OR render as a dedicated backbone tint when `loadBucket` is 0 | The near-full-mesh (~30 legs) must read as the inter-center highway |
| Spoke leg (light) | Thin, dimmer | stroke width **2px** (existing default), `opacity 0.55` | Spoke legs recede beneath the backbone + trailers |

**Tier source fields (added this phase, see CONTEXT Area 3):** hub DTO gains `kind`/`tier` (`"center" | "spoke"`);
route DTO gains `isBackbone: boolean`. The cached style functions branch on these FIRST (tier), then fall through
to the existing bucket logic — same precedence pattern as `hubStyle` (duty → volume → default) and
`routeStyle` (risk → load → default). Add a new pre-allocated `Style` per (tier × bucket) cell, or compose
tier-ring + bucket-fill at module load; never allocate per frame.

**Legend:** extend the existing legend single-source-of-truth arrays so the two hub tiers (large amber-ring =
"Regional center", small = "Spoke hub") and the two leg weights ("Backbone" heavy / "Spoke leg" light) are
labeled. The legend MUST be derived from the same constants the styles use (the `HUB_COLORS`/`*_LABELS` pattern).

### VIZ-15 declutter — cluster vs. individual zoom thresholds

OpenLayers 10 declutters all declutter-enabled Vector/VectorImage layers collectively by z-index. Use
`declutter: true` on a `VectorImageLayer` for the hub layer wrapped in an `ol/source/Cluster`. Trailers stay
on their existing `VectorLayer` (they animate via postrender; do NOT cluster animated trailers — clustering
would fight the in-place geometry tween). Use resolution thresholds, NOT hard zoom-level switches, so the
transition is smooth.

| State | Condition | Rendering |
|-------|-----------|-----------|
| Clustered | continental view (≈ zoom ≤ 5 / large map resolution) | Cluster bubble: slate-700 `#334155` disc, white count text, radius scales 14→22px with member count (log-bucketed into ≤4 cached sizes — never per-cluster alloc). Regional centers always render individually (a center is never absorbed into a spoke cluster — filter centers out of the clustered source, keep them on their own un-clustered tier-1 layer). |
| Decluttered / individual | zoomed-in (≈ zoom ≥ 7) OR cluster member-count == 1 | Individual hub markers with their tier + bucket style; OL `declutter: true` drops overlapping glyph/text labels so labels never pile up. |
| Transition | between thresholds | Cluster `distance: 40` / `minDistance: 20` governs grouping; `VectorImageLayer` renders as an image for cheap pan/zoom at 100+ features. |

**Clutter budget:** at the default continental zoom 4, the map must read cleanly with 80–130 hubs + ~30 backbone
legs + live trailers. Centers (≤8) always visible and labeled; spokes cluster; spoke leg labels suppressed via
declutter; only the backbone + centers carry standing labels.

and MUST be reused verbatim (single source of truth, used by both the existing AlertFeed COORD-03 row and the new
suggestion feed). Suggestion `kind` ∈ `reroute | hold | consolidate | dispatch`; reject `reasonCode` ∈
`hos | fuel | dock | infeasible`.

| Element | Copy |
|---------|------|
| Primary CTA (toggle) | **"Suggestions"** — a toggle control (label `Suggestions`, ON shows blue accent). Tooltip/aria-label: "Show advisory coordination suggestions on the map" |
| Section title (feed) | **"Advisory Suggestions"** (uppercase per `.right-rail__section-title`); count badge mirrors the Live Exceptions count badge |
| Suggestion kind labels | `reroute` → **"Reroute"**, `hold` → **"Hold"**, `consolidate` → **"Consolidate"**, `dispatch` → **"Dispatch"** (Title Case, mirror `kindLabel()` in AlertFeed) |
| Accepted row body | **"Accepted — {kind} {entityId} → {toHubId}"** (green); e.g. "Accepted — Reroute T-0042 → DFW" |
| Rejected row body | reuse `COORDINATION_REJECT_LABELS[reasonCode]` verbatim: **"won't divert: HOS"**, **"won't divert: fuel"**, **"won't dispatch: dock full"**, **"declined: infeasible"** (red) |
| Empty state heading | **"No active suggestions"** (mirror `.alert-feed__empty` "No active exceptions") |
| Empty state body | (none — single-line empty state, matching AlertFeed's single-line pattern) |
| Toggle-OFF state copy | When the toggle is OFF the map overlay + feed are hidden; the toggle reads `Suggestions` (no badge). No separate empty message needed. |
| Error state | Suggestions ride the existing ws envelope; on disconnect the feed simply stops updating (same as AlertFeed). No dedicated error string — degrade silently like the map's stubbed-geo fallback. |
| Destructive confirmation | **None.** This overlay is read-only/advisory — accept (green) and reject (red) are SERVER-EMITTED outcomes the operator observes, not actions the operator takes. There is NO operator-initiated destructive action in this phase, so no confirmation dialog. |

### VIZ-17 suggestion overlay — visual + interaction contract

- **Opt-in toggle:** a small toggle control (place near the map, e.g. a corner overlay control or a rail header
  control consistent with `SpeedControl`). Default **OFF** (clean map first; suggestions are an opt-in layer).
  ON → blue accent (`#1d4ed8`), the map suggestion layer becomes visible AND the "Advisory Suggestions" rail feed
  populates. The same toggle governs both surfaces.
- **Accept = green / reject = red:** accepted suggestions render green (`#16a34a` marker fill / `#4ade80` feed text,
  matching the existing onTime/win greens); rejected render red (`#dc2626` marker / `#f87171` feed text, matching
  the existing high-severity/loss reds). These are the ONLY two outcome colors — do not introduce a third.
- **Map overlay marker:** a transient Point marker at the target trailer's (or target hub's) location, radius **13px**,
  green or red fill, white 2px stroke, glyph `✓` (accept) / `✕` (reject) drawn via OL `Text` (cached style per
  outcome — two pre-allocated `Style`s, zero per-frame alloc, mirror `inductionColoring.ts`). Decluttered:
  `declutter: true` on the suggestion layer so a burst of suggestions never stacks into an unreadable pile.
- **Transient lifetime:** suggestions are tick-only (new optional `suggestions?` field on `TickPayload`, mirroring
  `inductionEvents`); a map marker flashes for **~2500ms** then is removed individually (reuse the
  `flashInduction`/`flashDelivery` add-then-`setTimeout`-remove pattern; `Date.now()`/`Math.random()` for marker-id
  uniqueness only, which is the established sanctioned use). The rail feed retains entries with the same
  `MAX_FEED_ENTRIES = 200` cap and newest-first sort as AlertFeed (clone `useAlertFeed` → `useSuggestions`).
- **Feed row styling:** clone `.alert-feed__entry` exactly; left-border + tint = green for accepted, red for rejected
  (reuse the `--low` green and `--high` red border/tint values). Reuse the kind-label + reason + action three-line
  layout. No `dangerouslySetInnerHTML` (threat T-05-16) — all text via React default escaping.
- **Consistency rule:** the suggestion feed and the Live Exceptions feed must be visually siblings — same paddings,
  same type sizes, same badge, same empty-state treatment. A glance should read them as the same component family.

## Assumptions & Notes (researcher discretion, per CONTEXT Area 3 "Claude's Discretion")

Because this is incremental viz on an existing design system with decisions captured in `27-CONTEXT.md`, the
following pragmatic, consistent-with-existing choices were made and are flagged for the planner/executor:

1. **Cluster `distance: 40` / `minDistance: 20` (px)** — chosen so adjacent metro hubs in dense corridors (NE,
   CA) group at zoom 4 but separate by zoom ~7. Tunable; if the empirical continental run shows over-clustering of
   distinct metros, lower `distance` toward 30.
2. **Centers excluded from the clustered source** — kept on their own un-clustered Tier-1 layer so a regional center
   is always individually visible/labeled (a center absorbed into a spoke cluster would break the VIZ-16 hierarchy).
3. **Trailers are NOT clustered** — they animate via the existing postrender tween; clustering animated points fights
   the in-place geometry update. VIZ-15's clutter target is hubs (the 80–130 static field), which is where clutter lives.
4. **Tier encoded by size + ring, not hue** — preserves every existing metric color encoding (volume/load/risk/duty).
5. **Suggestion overlay default OFF** — "clean map first" is the stated VIZ-15 goal; the advisory overlay is opt-in.
6. **~2500ms suggestion flash** — slightly longer than the 2000ms induction/delivery flash so an operator can read the
   accept/reject outcome during a live demo. Tunable.
7. **Reject copy reused verbatim from `COORDINATION_REJECT_LABELS`** — single source of truth; the COORD-03 "won't
   divert: HOS/fuel" headline moment must read identically in the exception feed and the suggestion feed.
