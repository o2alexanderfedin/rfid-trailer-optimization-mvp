## Findings by Severity

### HIGH
_None._

### MEDIUM
_None._

### LOW

1. **Timing-config plumbing asymmetric (optimizer).** `server.ts` forwards `deps.timing` to the simulator (controls random draws) but NOT to the optimizer: `RollingLoop` is built with `buildTwinSnapshot(snapshotDb)` (no timing arg → `DEFAULT_TIMING_CONFIG`) and `runEpoch`'s `input.timing` is never set on the live path. Production `main.ts` passes no timing, so both sides use the default and agree — harmless as shipped. Latent DRY break: if a deployment ever set `deps.timing` non-default, the simulator would draw from it while the optimizer kept planning against `DEFAULT_TIMING_CONFIG`, silently desyncing. Recommend threading `deps.timing` through `RollingLoop`/`buildTwinSnapshot`, or documenting the demo as fixed to the default config.

2. **"Config change → different plan" proven only at the pure-core boundary (optimizer).** The OPT-09 sub-clause is exercised by injecting `input.timing` into `runEpoch` directly and pinning the `buildTwinSnapshot` timing arg; there is no end-to-end test driving a non-default timing config through the live server, because (per finding 1) the live `RollingLoop` has no timing input to vary. Default-path realistic behavior is fully tested.

3. **Center-dwell tests use a pinned timing override (sim-timing).** `center-dwell.unit.test.ts` necessarily pins a `timing: PINNED` config to make the spoke/center gap exactly readable, so it asserts wiring under a test-only injection rather than the literal `DEFAULT_TIMING_CONFIG`. Appropriate test design; default-path firing of `dwellCenter` is independently confirmed empirically (avg gap 95.8 vs spoke-only ~27) and structurally, and the exact default values are locked by `domain/timing.test.ts`.

4. **Cross-country transit medians exceed short integration horizons (sim-timing).** Large per-leg ORS durations (e.g. MEM-SEA ~2249-3000+ min) mean short-horizon integration tests can't see a full round trip — which is precisely why the documented DIP `timing` override exists for lifecycle tests. Confirmed test-only: `main.ts` never sets `deps.timing`, and every live `timing:` reference is a conditional spread omitting the field when undefined.

5. **API-boundary test weaker than the requirement (viz).** `queries.int.test.ts:172` only asserts `r.geometry.length > 1` for `GET /routes`, which a 24-point great-circle fallback would also satisfy. The strong road-vs-great-circle assertion exists only at the sim-unit level (`buildRoutes(USA_HUBS)`). Behavior is correct and live-wired, but a future regression silently reverting the API to great-circle would not be caught at the API boundary. Recommend tightening the `/routes` assertion to check a known multi-vertex leg (e.g. `route-MEM-SEA` has 175 pts > 24).

6. **Great-circle fallback confirmation, not a defect (viz).** The absent-file branch is the only path that exercises great-circle in tests; committed-file presence makes the road path the default. File is committed in both `src/` and `dist/`, dist byte-identical — no dark/stale-default risk. Checksum drift guard protects against future hub moves silently producing stale geometry.

7. **No committed secret (hardening).** The `git grep eyJ` hit is a FALSE POSITIVE inside a base64 sha512 integrity hash in `pnpm-lock.yaml` (not a JWT). No ORS API key in any tracked file; `.env` is untracked (only `.env.example` tracked). The real ORS key lives in local untracked `.env`, used only by `scripts/precompute-routes.ts` at precompute time — never shipped to the runtime hot path.

8. **Determinism keystone intact (hardening).** `projections-golden-replay.int.test.ts` builds the twin live from a seeded stream, then TRUNCATEs + replays `readAll(0n)` by `global_seq` and asserts byte-identical serialization (+ deep-equal, + second-rebuild-identical). Deliberately non-monotonic `occurredAt` offsets prove ordering is by `global_seq`, not wall-clock — catching any leaked `Date.now`/`Math.random`/unstable-sort.

9. **Traceability complete, cosmetic doc drift (hardening).** `REQUIREMENTS.md` maps all 7 reqs to phases (OPT-09/10→P7, TIME-01/02/VIZ-06→P6, HRD-01/QA-01→P8). All per-phase SUMMARYs are marked COMPLETE & verified, but the REQUIREMENTS traceability Status column still reads "Planned" — cosmetic staleness only, not a wiring gap.

## Verdict

All seven v1.1 requirements (OPT-09, OPT-10, TIME-01, TIME-02, VIZ-06, HRD-01, QA-01) are **genuinely live-wired on the production demo path and backed by behavior-asserting tests** — none are "dark but tested." The demo entrypoint (`main.ts → buildServer / driveSimulationPaced`) drives the realistic log-normal time model by default: real ORS drive times for all 18 legs (not flat `TRANSIT_MIN=30`), role-keyed center-vs-spoke dwell drawn exactly once per turnaround via the single shared `expectedMinutes` estimator (DRY across simulator and optimizer), committed road polylines rendered on the OpenLayers map, a tolerant speed fallback built into the live `parseEnvelope`, and four socket behaviors verified over a real MSW WebSocket through the production `WsProvider`. The prior "tested-but-dark" trap (orphaned `useWsEnvelope` hook) was already excised in `18c68ce` and re-verified gone, and the golden-replay determinism keystone remains byte-identical. The milestone does **not** qualify as fully "passed" only because of nine LOW, non-blocking debt items — chiefly the asymmetric timing-config plumbing (the optimizer's `RollingLoop` hardcodes `DEFAULT_TIMING_CONFIG` rather than honoring `deps.timing`, a latent DRY break that is harmless in the shipped demo), two unit-only-vs-end-to-end test boundaries (the "config change → different plan" clause and the `/routes` road-vs-great-circle distinction), and a cosmetic REQUIREMENTS "Planned" status that should read "Done." Status: **tech_debt**.
