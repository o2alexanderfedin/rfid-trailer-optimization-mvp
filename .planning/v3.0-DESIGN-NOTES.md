# v3.0 Design Notes — Continental, Agent-Based, Event-Driven Coordination Network

**Status:** DESIGN SEED (not yet a milestone). Captured 2026-06-26 from an interactive design
discussion. Feed this into `/gsd-new-milestone` (questioning → research → requirements → roadmap).

**One-line vision:** Grow the network to a realistic continental scale (hubs in big cities across
every state), and replace the single global rolling optimizer with **decentralized OODA agents**
(trucks + hubs) that raise events, plus **coordination-center entities** that observe those events and
**suggest actions** — all while preserving the event-sourced, deterministic, golden-replay foundation.

This direction emerged while fixing the v2.0 demo: the single global optimizer re-solved over growing
state each cycle and (together with an O(n²) projection fold, fixed in v2.1) caused the sim to grind to
a halt. A decentralized, event-driven, locally-bounded decision model is both more realistic AND more
scalable.

---

## Design elements (user-stated, verbatim intent preserved)

### 1. Auto-generate hubs in big cities — 1–3 per state
- "Automatically put hubs in all big cities (at least 1, and up to 3 per state)" → ~80–130 hubs.
- Data-driven: a curated big-US-cities dataset (name, state, lat/lon, population/rank) → pick the top
  1–3 metros per state. Deterministic, static (no clock/RNG) so it stays golden-reproducible.
- Today's seed is 10 IATA hubs in `packages/simulation/src/network/hubs.ts`; this generalizes it.

### 2. Regional-center topology (DECIDED: "Regional centers")
- Today: a SINGLE center (Memphis), every hub a spoke on it (`buildRoutes` in
  `packages/simulation/src/network/routes.ts` centers on `USA_HUBS[0]`).
- v3.0: auto-pick a handful of **regional sort centers** (e.g., largest metro per region/timezone);
  each big-city hub is a spoke on its **nearest** center; centers form an inter-center **backbone**.
  Keeps per-center fan-out bounded (good for perf) and is realistic at 100+ hubs.
- Requires the engine to support **multiple centers** (today it assumes one) — a real change to
  `buildRoutes`, the freight-flow model (spoke→regional-center→backbone→regional-center→spoke), and the
  optimizer/twin scope.
- Route geometry (DECIDED: "Great-circle arcs") — compute great-circle arcs in-code for the many new
  legs (free, instant, deterministic); skip per-leg OpenRouteService road geometry at this scale.

### 3. OODA step-agents for trucks and hubs (DECIDED framing below)
- "Using OODA design for trucks and hubs makes sense, with step functions."
- **Adopted framing (determinism-preserving):** each truck/hub gets a deterministic `step()` that runs
  **Observe** (read projections / local state) → **Orient** (assess) → **Decide** (seeded, pure) →
  **Act** (emit domain events). OODA is *how commands are generated*; the **event log stays the source
  of truth**, reducers still fold, replay stays byte-identical for a given model+seed.
- **Rejected framing:** a full agent-based rewrite that replaces the event queue with per-tick agent
  loops — too high-risk to the event-sourced core + determinism. Keep event-sourcing; layer OODA on top.
- Step cadence: agents step on the existing deterministic tick/EventQueue (per-tick or per-N-tick),
  not wall-clock.

### 4. Coordination-center entities (newest element)
- "Add an entity (or entities) of coordination centers that will suggest actions based on trucks/hubs
  raised events."
- This is the **Orient/Decide layer as a first-class entity** — in event-sourcing terms a
  **process manager / policy**: a coordination center *subscribes to* truck/hub events, maintains its
  own localized state, and **emits suggestion/command events** (e.g. re-route, hold, consolidate,
  dispatch) that trucks/hubs then act on.
- Naturally **decentralized**: one coordinator per regional center (bounded scope) instead of one
  global optimizer over all-state — directly addresses the scaling + stall problems.
- Suggestions-vs-commands is an open design point: do coordinators *suggest* (advisory, an operator/
  agent accepts) or *command* (binding)? Likely start advisory (emit `ActionSuggested`), let agents
  accept → emit the binding event. Mirrors the existing optimizer "recommendation" model.

## Determinism keystone (non-negotiable, carries from v1.0–v2.0)
- Event-sourced; pure reducers; seeded RNG per substream; golden-replay.
- OODA decision logic **changes the event stream**, so v3.0 is a **NEW model with NEW goldens** (the
  v2.0 seed-42 golden `3920accc05220b45f79736cc98c9773fa7ffd8df08eb607bdbed2b8c054d6861` stays the
  baseline; the OODA/coordination model gets its own goldens). Gate every feature behind a flag so
  flags-off stays byte-identical to v2.0.

## How this resolves the problems we hit
- **Scaling to 100+ hubs:** regional centers + local decisions (bounded fan-out + bounded per-agent
  cost) instead of one global star + one global solve.
- **The freeze:** decentralized per-agent/per-coordinator decisions are O(active), not O(total state).
  (v2.1 already fixed the O(n²) projection fold and made the optimizer fire-and-forget; v3.0 removes the
  global-solve scaling pressure structurally.)
- **Realism:** autonomous hubs/trucks + regional coordinators reacting to local conditions.

## Open questions for milestone questioning (`/gsd-new-milestone`)
1. Big-city dataset source + "big city" ranking (population? metro? a fixed curated list?).
2. How many regional centers, and the region/timezone partition + nearest-center assignment rule.
3. Backbone topology between centers (full mesh? ring? hub-of-hubs?).
4. OODA step cadence (every tick vs every N) and per-agent decision scope.
5. Coordinator output: advisory `ActionSuggested` (accept→act) vs binding commands. How does it relate
   to / replace the existing rolling optimizer?
6. Migration: does v3.0 replace the single-center model or coexist behind a flag? New goldens plan.
7. Perf budget: target sustained sim-min/wall-sec at 100+ hubs; finish the `twin-snapshot` incremental
   follow-up (it still does full event-log scans per optimizer epoch).
8. Viz: rendering 100+ hubs + regional backbones + suggestion overlays without clutter.

## Carry-over tech debt (from v2.0 / v2.1, relevant here)
- `twin-snapshot.ts`: two full event-log scans per optimizer epoch (O(log)/O(log²) over a run) — make
  the snapshot builder fold incrementally from a cursor. (Bounded by `optimizerEveryTicks`; secondary.)
- OUT-05 delivery-KPI vs CONT-04 retention: materialize the delivery-KPI projection before enabling
  event-log retention in a long run (see `milestones/v2.0-MILESTONE-AUDIT.md`).
- Continuous-mode demo wiring + per-frame DB round-trip batching (residual ~40ms/event keyed upserts).
