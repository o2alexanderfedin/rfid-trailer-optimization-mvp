## Cross-phase integration

### Chain A — Driver HOS end-to-end (domain → sim → projection → endpoint → ws → UI): CONFIRMED

`domain events` (`@mm/domain` 7 new events, closed union, `contract.assert.ts` green)
→ **sim emits them** (`engine.ts`, gated on `hosOn`; live demo turns it ON)
→ **projection folds them** (`driverStatusReducer` → `driver_status` row, HOS fields via reused
   Phase-10 engine, keyed off `occurredAt`)
→ **endpoint returns them** (`GET /api/hubs/:id/detail` joins `trailer_state.driver_id` →
   `driver_status`)
→ **ws carries buckets** (`driverBucketsPerHub` → `HubState.driverCount/onBreakCount/restingCount`)
→ **UI renders** (`HubDetail.tsx` rows + `dutyColoring.ts` map markers).

The live-demo toggle is REAL and reaches the sim engine:
`packages/api/src/main.ts:86` `resolveDemoHosEnabled()` (default ON) →
`:94-101` `driveSimulationPaced({…, hosEnabled, hosConfig: DEFAULT_HOS_CONFIG})` →
`packages/api/src/sim/driver.ts:461-462` spreads `hosEnabled`/`hosConfig` into `simulate(…)` →
`packages/simulation/src/engine.ts:246` `hosOn = hosEnabled === true` gates all driver emission.
Verified: with HOS on, the engine seeds 15 drivers, assigns per trip, accrues minutes,
parks/relays on breach, and emits the driver + load/unload events, so `driver_status` is
populated for the panel + map. **This chain is intact end-to-end.**

### Chain B — Optimizer HOS (`driver_status` → twin-snapshot → restCost → hard gate → relay): BROKEN at the hard-gate link

`driver_status` → **twin-snapshot** (`twin-snapshot.ts:325` reads `driver_status`) →
soft **restCost** (OPT-HOS-01: `restPenaltyFor` → objective; LIVE, default weight 0) →
**hard gate** (OPT-HOS-02) → **insertRest/relay** (OPT-HOS-03).

The break is between twin-snapshot and the hard gate. `twin-snapshot.ts:381-383` builds the
driver as exactly `{ driverId, remainingDriveMinutes }` — it never sets the optional
`hosClock`/`hosConfig` on `TwinDriver`. The hard gate is activated ONLY when a full `hosClock`
is present (`epoch.ts:302-308` `driverHosContextFor` → `undefined` whenever `hosClock` is
absent). Therefore on the live path `route.hosFeasible` stays `undefined`, the gate's verdict
is never `false`, and `firstHosInfeasibleLeg` (OPT-HOS-03) is never invoked
(`epoch.ts:530-533`). The whole hard-enforcement + recommendation branch is unreachable in the
running optimizer.

**Why the data is missing:** the `driver_status` projection (`schema.sql:53`) persists only the
*derived* HOS summary (`remaining_drive_minutes`, `duty_window_deadline`, `total_driven_minutes`,
`weekly_on_duty_min`) — NOT the full `HosClock` (no `drive_today_min` / `since_last_break_min` /
`come_on_duty_at` / sleeper accumulators). The full clock IS available in the event log
(`DriverDutyStateChanged.payload.clock`) but is never carried into the read model, so
twin-snapshot has nothing to attach. Both my direct read and an independent verification agent
reached this conclusion from the source.

**No-op stub scan (the other dark-requirement class): CLEAN.** The driver/load-unload events are
classified as **scope-neutral** `return []` in `optimizer/src/rolling/scope.ts:57-63` — this is
*correct by design* (the optimizer consumes driver state via the twin snapshot, not via
event-scoping), not a missed fill-in. In `@mm/projections`, the reducers that should no-op on
driver events do (`audit-timeline`, `exceptions`, `hub-inventory`, `zone-estimate`,
`tag-registry`, `geo-track`, `package-location`), while the two that Phase 13 was supposed to
make real ARE real: `trailer-state.ts:124,139` stamp `driver_id` on `DriverAssignedToTrip` /
`DriverSwappedAtHub`, and `driver-status.ts` / `driver-assignment.ts` fold the full driver
lifecycle. No driver event is silently dropped where a later phase should have handled it.

## Gaps / Tech debt

### GAP-1 — OPT-HOS-02 (hard HOS gate) is DARK on the live optimizer path  [milestone-blocking]

- **Evidence:** `packages/api/src/optimizer/twin-snapshot.ts:381-383` constructs
  `TwinDriver` as `{ driverId, remainingDriveMinutes }` and never sets `hosClock`/`hosConfig`.
  `packages/optimizer/src/rolling/epoch.ts:304` `driverHosContextFor` returns `undefined`
  whenever `hosClock` is absent, so `routeTrailers` is called without a `driver` context
  (`epoch.ts:521`) and `route.hosFeasible` stays `undefined`. The gate logic
  (`route-trailers.ts` `hosLegsFeasible`, `feasibility.ts` `restMin` fold) is correct and
  unit-tested, but never reached by the running RollingLoop
  (`server.ts:142 buildSnapshot → live-loop.ts:111 → runEpoch`).
- **Root cause:** the `driver_status` projection (`schema.sql:53`) stores only the derived HOS
  summary, not the full `HosClock` the gate needs. The clock exists in the event log
  (`DriverDutyStateChanged.payload.clock`).
- **Fix (recommended):** extend `driver_status` to persist the full `HosClock` snapshot (add the
  clock columns, fold from `DriverDutyStateChanged` in `driverStatusReducer`), then in
  `twin-snapshot.ts` attach `hosClock` (+ `hosConfig`) to the `TwinDriver`. ~1 reducer field
  set + 1 table change + 1 snapshot read. This flips the existing, already-tested gate live.

### GAP-2 — OPT-HOS-03 (insertRest / relay recommendation) is DARK on the live path  [milestone-blocking, dependent on GAP-1]

- **Evidence:** `packages/optimizer/src/rolling/epoch.ts:530-533` only calls
  `firstHosInfeasibleLeg` when `driverCtx !== undefined && route.hosFeasible === false` — both
  impossible live until GAP-1 is fixed. `repair/local-repair.ts` `hosVariants` (insertRest/relay)
  is implemented and unit-tested but never surfaced through `EpochRecommendation` on the live
  path.
- **Fix:** resolved automatically by GAP-1 (it shares the same missing `hosClock` data). No
  separate change needed once the hard gate receives a real clock.

### Note (not a gap) — OPT-HOS-01 is default-neutral by design

`DEFAULT_OBJECTIVE_WEIGHTS.restCost = 0`, so the soft preference is computed and folded every
epoch but contributes 0 to the objective unless an operator raises the weight. This is the
intended back-compat default (prior plans reproduce byte-identically) and is documented; flagged
here only so the "soft awareness is live but neutral" nuance is explicit.

### Recommendation for milestone closure

The milestone's **core demonstrable value** (driver HOS enforced in the simulation; Hub Detail
panel showing live driver duty + remaining legal drive time; map duty coloring) is fully
delivered and live. The two gaps are confined to the *optimizer's* hard enforcement. Either:
(a) close GAP-1 (small, well-scoped — persist the clock + attach it in the snapshot) to make the
milestone goal's "fully enforced in both the simulation and the optimizer" literally true; or
(b) explicitly re-scope OPT-HOS-02/03 to "optimizer-aware (soft) live; hard enforcement
unit-proven, live-wiring deferred" and record it as carried debt. Do **not** mark the milestone
DONE-as-specified without one of these, since the REQUIREMENTS goal text claims live optimizer
enforcement.
