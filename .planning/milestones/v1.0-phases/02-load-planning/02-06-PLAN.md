<objective>
Expose the load planner as ONE thin, read-only API endpoint (`POST /plan`) so Phase 2 is demoable:
accept a set of packages + a route (in the request body), run the pure pipeline
(`aggregate → planLoad + baselinePlan → validatePlan → scorePlan → instructions → planExplanation`),
and return `{ plan, baseline, instructions, validation: { hardViolations, softViolations }, scores:
{ rehandleScore, utilizationScore }, explanation }`. The endpoint GATES on feasibility — it exposes
the validation verdict and only treats a plan as shippable when `isFeasible` is true (P2 held at the
boundary). No plan persistence / `PlanGenerated` events (deferred to Phase 4).

Purpose: the correctness lives in the pure modules; this plan is the thin composition root that turns
them into a thing a human can call and eyeball (the dock-worker card + rationale). It is the MVP
capstone of the phase — after it lands, the planner is demoable end to end.

Output: a `POST /plan` route registered in `@mm/api`, wired to the three pure packages, with an
`app.inject()` integration test covering the happy path and the infeasible-gating path.
</objective>
