<objective>
Complete the planner: **rehandle + utilization scoring** (LOAD-06/07, computed AFTER the hard gate and
returned as a `ScoreResult` distinct from `FeasibilityResult` — P2), **human-readable loading
instructions** by nose/middle/rear zone (LOAD-08), **plain-English per-placement rationale** plus a
plan-level explanation (LOAD-10), and the **naive FIFO baseline planner** (LOAD-09) that runs on the
SAME inputs and flows through the SAME scoring/KPI plumbing — so the Phase-5 "money slide" is wiring,
not a rebuild (P8). Capped by the baseline-vs-optimizer test: on a blocking-prone scenario the
optimizer's rehandle score must be ≤ the baseline's (it has something to beat, and beats it).

Purpose: scoring is the soft layer that only runs once feasibility passes. Sharing one `scorePlan`
plumbing between `planLoad` and `baselinePlan` is the structural guarantee that the later before/after
comparison is apples-to-apples. The rationale turns scoring internals into the explainability the demo
sells. After this slice, a user gets a feasible, scored, explained plan AND a baseline to measure it
against.

Output: `scoring.ts`, `instructions.ts`, `rationale.ts`, `baseline.ts`, and the baseline-vs-optimizer
test — all in the existing `@mm/load-planner` package.
</objective>
