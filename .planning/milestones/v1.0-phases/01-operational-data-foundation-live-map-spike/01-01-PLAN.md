## Phase Goal

**As a** logistics operator (and demo audience), **I want to** an auditable, deterministically-replayable operational twin fed by a simulator and lit up on a live USA map, **so that** I can ask where any package or trailer was and watch the network move in realtime.

> Note (planner): the ROADMAP `**Goal:**` line for Phase 1 is outcome-shaped prose, not the canonical "As a / I want to / so that" form. The story above was reconstructed faithfully from that goal + 01-CONTEXT.md for MVP-mode framing. Consider running `/gsd mvp-phase 1` if a canonical ROADMAP goal line is desired.

<objective>
Stand up the monorepo and prove the entire Phase-1 spine end-to-end with the thinnest possible vertical slice: a real Postgres event append -> one inline projection -> one API read -> a React/OpenLayers web app rendering one real hub from that read, with one passing integration test against a real Postgres container on OrbStack.

This is the **Walking Skeleton**. It deliberately implements almost nothing of the domain — it exists to nail the architectural decisions (monorepo layout, strict TS, OrbStack/Testcontainers, Fastify, OpenLayers+OSM, downward-only package deps) so every later plan in this phase fleshes out a package whose contracts and wiring already exist. It also produces `SKELETON.md` recording those decisions.

Purpose: De-risk the whole stack in one slice before any deep domain work; give every Wave-1+ plan a working scaffold and clear file-ownership boundaries to minimize cross-task contention under rival execution.
Output: Working monorepo; docker-compose Postgres on OrbStack; Testcontainers fixture; one green integration test; a live OSM map with one real hub; `SKELETON.md`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/web -> api (HTTP) | The web app calls GET /hubs; untrusted query input crosses here (read-only in Phase 1). |
| sim/ingestion -> event-store (append) | Event payloads cross into durable storage; must be typed/validated (full enforcement in Plan 02). |
| OSM tile CDN -> web | Third-party tiles loaded into the map. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | events table | mitigate | Append-only schema (no UPDATE/DELETE path in code); `UNIQUE(stream_id, version)` from the start; full enforcement Plan 03. |
| T-01-02 | Information disclosure | GET /hubs | accept | Public hub geo only; no PII in a simulation demo. |
| T-01-03 | Tampering | OSM tiles over network | mitigate | Load tiles over HTTPS, set `crossOrigin: 'anonymous'`; pin OpenLayers version. |
| T-01-04 | Denial of service | OSM tile provider | accept | Demo-scale traffic; respect OSM tile usage policy (documented in README); local tile cache deferred. |
</threat_model>
