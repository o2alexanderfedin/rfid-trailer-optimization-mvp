<objective>
Create the PURE `@mm/sensor-fusion` module's fusion engine: RSSI→likelihood (capped), per-dwell
observation windowing, and rule-based Bayesian zone fusion with a Markov zone-transition prior and an
entropy floor. This is the OBSERVED-evidence engine of the two-layer (planned vs observed) design, and
it is where the anti-P5b defense lives: repeated reads of one tag in one dwell can NEVER drive confidence
to 1.0.

Purpose: RFID RSSI badly violates the Bayesian independence assumption (multipath/interference) — treating
N dependent reads as N independent updates causes overconfident lock-on (P5b). The Google AI Mode consult
(03-RESEARCH.md) gives concrete, adopt-now techniques: likelihood cap ≤ 0.85, per-dwell aggregation using
90th-percentile RSSI + read-rate density, an entropy floor, and a Markov transition prior. All deterministic
and config-driven.

Output: a pure, fully unit/property-tested fusion engine producing a confidence-scored zone estimate
(spec §8.4: `{ packageId, trailerId, estimatedZone, confidence, lastReliableCheckpoint, lastObservedAt }`).
</objective>

  <behavior>
    likelihood.ts — `rssiToLikelihood(rssi, readerType, config)`:
      - MONOTONIC: higher rssi ⇒ higher (or equal) likelihood, for a fixed readerType.
      - Reader-type priors: a `dock-portal` reader is high-reliability; a `trailer-antenna` reader is
        zone-ish (lower reliability). Encoded as config-supplied type weights.
      - CAPPED: output ≤ `config.maxLikelihood` (default 0.85) and ≥ a small floor (> 0). A single strong
        RSSI can never yield 1.0.

    window.ts — `windowObservations(reads, config)`:
      - Groups raw reads by key `(tagId, readerId, dwellWindowId)` and emits ONE `WindowedObservation`
        per group, NOT one per read (anti-P5b collapse).
      - Aggregated RSSI = the 90th-percentile (or mode) of the group's RSSI values — NOT the mean
        (multipath drops skew the mean; per the AI Mode consult).
      - Carries `readCount` (read-rate density): the aggregate's effective confidence weight factors read
        count (high RSSI w/ 1 read < moderate RSSI w/ 40 reads), but the per-window likelihood remains
        bounded by maxLikelihood regardless of readCount.
      - Deterministic ordering: groups emitted sorted by their key (no Map-iteration-order dependence).

    fuse.ts — `fuseZone(prior, windowedObs[], config)`:
      - Bayesian posterior over {rear, middle, nose} starting from `prior` (a zone distribution).
      - Applies a Markov zone-transition prior: near-zero probability for impossible jumps
        (rear→nose without passing middle) before each update.
      - Blends an entropy floor (1–5%, config) each step so no zone probability reaches 1.0.
      - Returns a `ZoneEstimate` with `estimatedZone` = argmax zone, `confidence` = that zone's posterior
        probability (STRICTLY < 1.0, ≤ ceiling), plus `lastReliableCheckpoint` and `lastObservedAt`
        derived from the observations (no wall clock).
      - Empty `windowedObs` ⇒ returns the prior-derived estimate unchanged (a building block for "absence
        changes nothing"; absence-as-non-exception is enforced in Plan 04's detection, not here).
  </behavior>
